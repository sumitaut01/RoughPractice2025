üîê Authentication vs Authorization

üÜî Authentication (AuthN)

What: "Who are you?"

Purpose: Verifying the identity of a user or system

Example: Login with username/password, proving you are who you claim to be

Result: Confirms identity (valid/invalid credentials)

üîë Authorization (AuthZ)

What: "What can you do?"

Purpose: Determining what resources/actions an authenticated user can access

Example: Admin can delete users, regular user cannot

Result: Grants or denies access to resources

üí° Remember: Authentication comes FIRST, then Authorization. You can't authorize someone whose identity you haven't verified!

Real-World Example

// Step 1: AUTHENTICATION - User proves identity
String token = given()
    .contentType(ContentType.JSON)
    .body("{ \"username\": \"john\", \"password\": \"secret123\" }")
.when()
    .post("/api/login")
.then()
    .statusCode(200)
    .extract().path("token");

// Step 2: AUTHORIZATION - Check what user can access
// John is a regular user, trying to access admin endpoint
given()
    .header("Authorization", "Bearer " + token)
.when()
    .delete("/api/admin/users/123")
.then()
    .statusCode(403); // Forbidden - Authenticated but not Authorized

// John accessing his own profile - Authorized
given()
    .header("Authorization", "Bearer " + token)
.when()
    .get("/api/users/john/profile")
.then()
    .statusCode(200); // Success - Both Authenticated and Authorized

Common HTTP Status Codes

Status Code Scenario Meaning
401 Unauthorized Authentication Failed Invalid credentials or missing token
403 Forbidden Authorization Failed Valid credentials but insufficient permissions
200 OK Success Both authenticated and authorized

üåê OpenID Connect (OIDC)

OpenID Connect is an identity layer built on top of OAuth 2.0. It allows clients to verify user identity and obtain basic profile information.

Key Concepts

OAuth 2.0

Purpose: Authorization framework

Use case: "Let this app access my Google Drive"

Returns: Access tokens for resource access

OpenID Connect

Purpose: Authentication + Authorization

Use case: "Log in with Google"

Returns: ID token (who you are) + Access token (what you can do)

OIDC Flow Example

// Step 1: Get ID Token and Access Token from OIDC provider
Response authResponse = given()
    .formParam("grant_type", "password")
    .formParam("client_id", "my-app")
    .formParam("client_secret", "secret")
    .formParam("username", "john@example.com")
    .formParam("password", "password123")
    .formParam("scope", "openid profile email")
.when()
    .post("https://auth-provider.com/oauth/token")
.then()
    .statusCode(200)
    .extract().response();

String idToken = authResponse.path("id_token"); // Contains user identity info
String accessToken = authResponse.path("access_token"); // For API access

// Step 2: Use Access Token to call protected API
given()
    .header("Authorization", "Bearer " + accessToken)
.when()
    .get("/api/protected-resource")
.then()
    .statusCode(200);

ID Token vs Access Token

Aspect ID Token Access Token
Purpose Prove user identity Access protected resources
Format Always JWT Can be JWT or opaque string
Audience Client application Resource server/API
Contains User claims (name, email, sub) Permissions/scopes
Validation Validated by client Validated by resource server

üîç Interview Tip: When asked about OIDC, mention that it adds an identity layer to OAuth 2.0. OAuth 2.0 is for authorization ("can I access this?"), while OIDC adds authentication ("who am I?"). Together they provide both identity verification and access control.

üìä Validating Large/Complex Responses

When dealing with huge responses with many fields to validate, you have several approaches. Let's compare them:

Method Comparison

Approach Best For Pros Cons
JsonPath Selective field validation Fast, lightweight, no POJO needed No compile-time safety, string-based paths
Deserialization Complete object validation Type-safe, reusable POJOs, better maintainability Requires POJO creation, more setup
JSON Schema Structure validation Validates entire structure, reusable schema Doesn't validate actual values
Hybrid Best of both worlds Flexibility + type safety More code to write
‚úÖ Recommended Approach: Use Deserialization for large responses with many fields. Use JsonPath for quick spot checks or when you only need to validate a few fields.

üéØ Approach 1: JsonPath (Quick & Selective)

Use when: You need to validate specific fields or a subset of the response

@Test
public void testWithJsonPath() {
    // For small validations - quick and dirty
    given()
    .when()
        .get("/api/users/1")
    .then()
        .statusCode(200)
        .body("id", equalTo(1))
        .body("name", equalTo("John Doe"))
        .body("email", containsString("@"))
        .body("address.city", equalTo("New York"))
        .body("address.zipCode", equalTo("10001"))
        .body("orders.size()", greaterThan(0))
        .body("orders[0].status", equalTo("completed"));
}

// Extract first, then validate - better for complex logic
@Test
public void testWithJsonPathExtraction() {
    Response response = get("/api/users/1");
    JsonPath json = response.jsonPath();
    
    // Extract and validate
    String name = json.getString("name");
    int age = json.getInt("age");
    List<String> hobbies = json.getList("hobbies");
    
    // Custom assertions
    Assert.assertEquals(name, "John Doe");
    Assert.assertTrue(age >= 18);
    Assert.assertTrue(hobbies.contains("reading"));
}

Pros & Cons

‚úÖ Pros

  • Quick to write
  • No POJO classes needed
  • Good for spot checking
  • Lightweight

‚ùå Cons

  • No compile-time safety
  • Hard to maintain for many fields
  • Typos in path strings
  • Verbose for large responses

üèÜ Approach 2: Deserialization (Recommended for Large Responses)

Use when: You need to validate many fields or the entire response structure

Step 1: Create POJO Classes

public class User {
    private int id;
    private String name;
    private String email;
    private Address address;
    private List<Order> orders;
    
    // Getters, setters, constructors
    // Can use Lombok: @Data, @Getter, @Setter
}

public class Address {
    private String street;
    private String city;
    private String zipCode;
}

public class Order {
    private int id;
    private String status;
    private double amount;
}

Step 2: Deserialize and Validate

@Test
public void testWithDeserialization() {
    // Deserialize response to POJO
    User user = given()
    .when()
        .get("/api/users/1")
    .then()
        .statusCode(200)
        .extract()
        .as(User.class);
    
    // Now validate with type safety
    Assert.assertEquals(user.getId(), 1);
    Assert.assertEquals(user.getName(), "John Doe");
    Assert.assertTrue(user.getEmail().contains("@"));
    
    // Nested object validation
    Assert.assertEquals(user.getAddress().getCity(), "New York");
    Assert.assertEquals(user.getAddress().getZipCode(), "10001");
    
    // List validation
    Assert.assertTrue(user.getOrders().size() > 0);
    Assert.assertEquals(user.getOrders().get(0).getStatus(), "completed");
    
    // Advanced validations
    double totalOrderAmount = user.getOrders().stream()
        .mapToDouble(Order::getAmount)
        .sum();
    Assert.assertTrue(totalOrderAmount > 100.0);
}

Step 3: Create Reusable Validation Methods

public class UserValidator {
    
    public static void validateUser(User user) {
        Assert.assertNotNull(user, "User should not be null");
        Assert.assertTrue(user.getId() > 0, "User ID should be positive");
        Assert.assertNotNull(user.getName(), "Name should not be null");
        Assert.assertTrue(user.getEmail().matches("^[A-Za-z0-9+_.-]+@(.+)$"), 
                         "Invalid email format");
    }
    
    public static void validateAddress(Address address) {
        Assert.assertNotNull(address);
        Assert.assertNotNull(address.getCity());
        Assert.assertTrue(address.getZipCode().matches("\\d{5}"));
    }
    
    public static void validateOrders(List<Order> orders) {
        Assert.assertNotNull(orders);
        Assert.assertFalse(orders.isEmpty());
        
        for (Order order : orders) {
            Assert.assertTrue(order.getId() > 0);
            Assert.assertNotNull(order.getStatus());
            Assert.assertTrue(order.getAmount() >= 0);
        }
    }
}

// Usage in test
@Test
public void testUserWithValidator() {
    User user = get("/api/users/1").as(User.class);
    
    UserValidator.validateUser(user);
    UserValidator.validateAddress(user.getAddress());
    UserValidator.validateOrders(user.getOrders());
}

Pros & Cons

‚úÖ Pros

  • Type-safe (compile-time checking)
  • Clean, readable code
  • Reusable POJOs and validators
  • Easy to maintain
  • IDE autocomplete support
  • Can use Java streams for complex logic

‚ùå Cons

  • Initial setup time (create POJOs)
  • Need to update POJOs when API changes
  • More classes to manage

üîÄ Approach 3: Hybrid (Best of Both)

Use when: You want flexibility with type safety

@Test
public void testHybridApproach() {
    Response response = given()
        .when()
            .get("/api/users/1")
        .then()
            .statusCode(200)
            .extract().response();
    
    // Quick validation with JsonPath
    response.then()
        .body("id", equalTo(1))
        .body("name", notNullValue());

        Assert.assertEquals(user.getLoyaltyStatus(), "GOLD");
    }
}

üìã Approach 4: JSON Schema Validation

Use when: You want to validate the structure without checking every value

Create Schema File (user-schema.json)

{
  "$schema": "http://json-schema.org/draft-04/schema#",
  "type": "object",
  "properties": {
    "id": { "type": "integer" },
    "name": { "type": "string" },
    "email": { 
      "type": "string",
      "pattern": "^[A-Za-z0-9+_.-]+@(.+)$"
    },
    "address": {
      "type": "object",
      "properties": {
        "city": { "type": "string" },
        "zipCode": { "type": "string" }
      },
      "required": ["city", "zipCode"]
    },
    "orders": {
      "type": "array"
    }
  },
  "required": ["id", "name", "email"]
}

Use in Test

@Test
public void testWithJsonSchema() {
    given()
    .when()
        .get("/api/users/1")
    .then()
        .statusCode(200)
        .body(matchesJsonSchemaInClasspath("schemas/user-schema.json"));
    
    // Then deserialize for value validation
    User user = get("/api/users/1").as(User.class);
    Assert.assertEquals(user.getName(), "John Doe");
}

üí° Decision Guide

Scenario Recommended Approach Reason
Quick smoke test JsonPath Fast to write, validates key fields only
1-5 fields to check JsonPath Lightweight, no overhead
10+ fields to validate Deserialization Type-safe, maintainable, reusable
Complex nested objects Deserialization Clean navigation, better readability
Business logic validation Deserialization Can use Java streams, custom methods
API contract testing JSON Schema Validates structure, reusable across tests
Data-driven testing Deserialization Compare POJOs easily
Dynamic response structure JsonPath Flexible, no rigid POJO needed
üéØ Best Practice: For production test suites with large responses:
  1. Use JSON Schema validation to ensure structure compliance
  2. Deserialize to POJOs for detailed field validation
  3. Create reusable validator utility classes
  4. Use JsonPath only for quick spot checks or dynamic fields

        .extract().path("token");

    // Step 2: Use token to create user
    String userId = given()
        .header("Authorization", "Bearer " + token)
        .contentType(ContentType.JSON)
        .body("{ \"name\": \"Test User\" }")
    .when()
        .post("/users")
    .then()
        .statusCode(201)
        .extract().path("id");

    // Step 3: Verify user was created
    given()
        .header("Authorization", "Bearer " + token)
        .pathParam("id", userId)
    .when()
        .get("/users/{id}")
    .then()
        .statusCode(200)
        .body("name", equalTo("Test User"));
}

üé§ Common Interview Questions

Q1: What is REST Assured?

Answer: REST Assured is a Java library that provides a domain-specific language (DSL) for writing powerful, maintainable tests for RESTful APIs. It simplifies API testing by providing methods to make HTTP requests and validate responses using a BDD (Behavior Driven Development) style syntax.

Q2: Explain Given-When-Then syntax

Answer:

  • Given: Preconditions - setup like headers, authentication, request body, parameters
  • When: Action - the actual HTTP method call (GET, POST, PUT, DELETE)
  • Then: Assertions - validations on response like status code, headers, body
Q3: How do you handle authentication in REST Assured?

Answer: REST Assured supports multiple authentication methods:

  • Basic Auth: auth().basic(username, password)
  • Preemptive Basic: auth().preemptive().basic()
  • Digest Auth: auth().digest(username, password)
  • OAuth 2.0: auth().oauth2(token)
  • Bearer Token: header("Authorization", "Bearer " + token)
Q4: What is RequestSpecification and ResponseSpecification?

Answer: These are reusable specifications that help avoid code duplication:

  • RequestSpecification: Defines common request settings like base URI, headers, authentication that can be reused across multiple tests
  • ResponseSpecification: Defines common response validations like status code, content type that can be applied to multiple tests
Q5: How do you validate JSON response?

Answer: JSON validation can be done using:

  • JsonPath: body("data.name", equalTo("John"))
  • Root level: body("name", equalTo("John"))
  • Nested fields: body("user.address.city", equalTo("NYC"))
  • Arrays: body("users[0].name", equalTo("John"))
  • Size: body("users.size()", is(10))
Q6: Difference between path() and extract()?

Answer:

  • path(): Directly extracts value from response body. Example: get("/users").path("name")
  • extract(): Provides more options like extracting entire response, specific path, headers, cookies. Example: extract().path("name") or extract().response()
Q7: How to handle file upload in REST Assured? 
given()
    .multiPart("file", new File("path/to/file.pdf"))
.when()
    .post("/upload");
Q8: What are Hamcrest matchers?

Answer: Hamcrest is a framework for writing matcher objects used in assertions. Common matchers include:

  • equalTo() - exact match
  • containsString() - substring match
  • hasItems() - collection contains items
  • greaterThan(), lessThan() - numeric comparison
  • notNullValue() - null check
Q9: How to perform JSON Schema validation? 
given()
.when()
    .get("/users")
.then()
    .assertThat()
    .body(matchesJsonSchemaInClasspath("schema.json"));

This validates that the response structure matches the defined JSON schema.

Q10: How to measure response time? 
// Assert response time
then()
    .time(lessThan(2000L), TimeUnit.MILLISECONDS);

// Extract response time
long responseTime = get("/users").time();
Q11: Explain Serialization and Deserialization

Answer:

  • Serialization: Converting Java object to JSON/XML format for request body. REST Assured automatically handles this when you pass POJO to body() method.
  • Deserialization: Converting JSON/XML response to Java object using as(Class) method.
Q12: How to handle cookies in REST Assured? 
// Set cookie in request
given()
    .cookie("session_id", "abc123")
.when()
    .get("/users");

// Get cookie from response
String cookie = get("/login").getCookie("session_id");
Q13: Explain the difference between Authentication and Authorization

Answer:

  • Authentication (AuthN): "Who are you?" - Verifying identity. Example: Login with username/password. HTTP Status: 401 Unauthorized when it fails.
  • Authorization (AuthZ): "What can you do?" - Determining permissions. Example: Admin can delete users, regular user cannot. HTTP Status: 403 Forbidden when it fails.
  • Key Point: Authentication must happen before authorization. You can't grant permissions to someone whose identity you haven't verified.
Q14: What is OpenID Connect (OIDC)?

Answer: OpenID Connect is an identity layer built on top of OAuth 2.0 protocol. Key points:

  • OAuth 2.0: Authorization framework - "Let this app access my Google Drive"
  • OIDC: Adds authentication - "Log in with Google"
  • ID Token: JWT containing user identity info (sub, name, email)
  • Access Token: For accessing protected resources/APIs
  • Use Case: When you click "Sign in with Google/Facebook", that's OIDC
Q15: How do you test 401 vs 403 scenarios? 
// 401 - Missing/Invalid token (Authentication failure)
given()
    // No auth header
.when()
    .get("/api/protected")
.then()
    .statusCode(401);

// 403 - Valid token but insufficient permissions (Authorization failure)
given()
    .header("Authorization", "Bearer " + userToken)
.when()
    .delete("/api/admin/users/123")
.then()
    .statusCode(403);
Q16: When to use JsonPath vs Deserialization for validation?

Answer:

  • Use JsonPath when:
    • Validating 1-5 fields only
    • Quick smoke tests
    • Dynamic or changing response structure
    • Spot checking specific values
  • Use Deserialization when:
    • Validating 10+ fields (RECOMMENDED)
    • Complex nested objects
    • Need type safety and compile-time checking
    • Reusability across multiple tests
    • Business logic validation (streams, calculations)
  • Best Practice: For large responses, use Deserialization with POJO classes and create reusable validator methods.
Q17: How do you validate a huge response with 50+ fields?

Answer: Best approach is to combine multiple strategies:

// Step 1: Schema validation for structure
.then()
    .body(matchesJsonSchemaInClasspath("user-schema.json"));

// Step 2: Deserialize to POJO
User user = response.as(User.class);

// Step 3: Use validator utility classes
UserValidator.validateBasicInfo(user);
UserValidator.validateAddress(user.getAddress());
UserValidator.validateOrders(user.getOrders());

// Step 4: Custom business logic
Assert.assertTrue(user.getTotalSpent() > 1000);

This approach provides: Structure validation (schema), Type safety (POJO), Reusability (validators), Maintainability

Answer:

Format Always JWT JWT or opaque
Audience Client app Resource server
Contains User claims (name, email) Scopes/permissions
