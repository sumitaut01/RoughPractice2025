JSON Web Token

Everything you need to understand JWT ‚Äî structure, flow, code, and common interview questions. Written for Java developers.

‚òÖ One-Line Answer for Interview

JWT is primarily used for Authentication + Authorization ‚Äî the server verifies WHO you are (authn) once, issues a token, and then uses that token to decide WHAT you can access (authz) on every subsequent request ‚Äî without hitting the database again.

Authentication vs Authorization ‚Äî The Classic Interview Trap
Concept Question it answers Example JWT role?
Authentication "Who are you?" Login with username + password JWT is issued here
Authorization "What can you do?" Can this user access /admin? JWT carries roles/claims

JWT sits at the boundary: you authenticate once (login), get a JWT, and then the JWT is used for authorization on every API call. The roles/permissions baked into the token tell the server what you're allowed to do.

JWT Structure ‚Äî 3 Parts Separated by Dots
Header (Base64URL)
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9 .
Payload / Claims (Base64URL)
eyJzdWIiOiJ1c2VyMTIzIiwibmFtZSI6IlJhaHVsIiwicm9sZXMiOlsiVVNFUiJdLCJpYXQiOjE2MzAwMDAwMDAsImV4cCI6MTYzMDA4NjQwMH0 .
Signature (HMAC / RSA)
SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

Header

Algorithm used to sign + token type. Example decoded:

{
  "alg": "HS256",
  "typ": "JWT"
}

Payload (Claims)

User data + metadata. Example decoded:

{
  "sub": "user123",
  "name": "Rahul",
  "roles": ["USER"],
  "iat": 1630000000,
  "exp": 1630086400
}

Signature

Ensures integrity. Nobody can tamper with the token. Computed as:

HMACSHA256(
  base64(header) + "." +
  base64(payload),
  secretKey
)

‚ö†Ô∏è Important: The payload is only Base64URL encoded ‚Äî NOT encrypted. Anyone can decode it. Never put passwords or sensitive data in the payload. The signature only proves the data wasn't tampered with.

Standard Claims You Must Know
sub Subject Who the token is about ‚Äî usually userId or email
iss Issuer Who created the token ‚Äî your auth service URL
aud Audience Who this token is intended for
exp Expiration Unix timestamp when the token expires ‚Äî ALWAYS set this!
iat Issued At When the token was created
jti JWT ID Unique ID for the token ‚Äî useful for revocation/blacklisting
How JWT Works ‚Äî The Full Auth Flow

User Logs In

Client sends POST /login with {username, password} ‚Üí Server validates credentials against DB.

Server Issues JWT

Auth server creates a JWT with user's id, roles, expiry, signs it with a secret key, returns it.

Client Stores Token

Frontend stores the JWT in memory or localStorage (mobile apps in secure storage).

Client Sends Token on Every Request

Each API call includes: Authorization: Bearer <token>

Server Validates JWT

Server verifies signature with the secret key + checks expiry. No DB call needed!

Access Granted or Denied

Server reads roles/claims from token to authorize the request, responds accordingly.

Java JwtUtil.java ‚Äî Generate Token
import io.jsonwebtoken.Jwts;
import io.jsonwebtoken.SignatureAlgorithm;
import java.util.Date;
import java.util.Map;

public class JwtUtil {

    private static final String SECRET_KEY = "mySecretKeyThatShouldBe32CharsLong";
    private static final long EXPIRY_MS = 86400000L; // 24 hours in ms

    public static String generateToken(String userId, String role) {
        return Jwts.builder()
            .subject(userId)               // "sub" claim
            .claim("role", role)            // custom claim
            .issuedAt(new Date())           // "iat" claim
            .expiration(new Date(
                System.currentTimeMillis() + EXPIRY_MS
            ))                               // "exp" claim
            .signWith(SignatureAlgorithm.HS256,
                SECRET_KEY.getBytes())     // sign with secret
            .compact();                     // build the token string
    }
}
Java JwtUtil.java ‚Äî Validate & Parse Token
    public static Claims validateAndParse(String token) {
        try {
            return Jwts.parser()
                .verifyWith(Keys.hmacShaKeyFor(SECRET_KEY.getBytes()))
                .build()
                .parseSignedClaims(token)
                .getPayload();
            // Returns Claims object ‚Äî expiry is auto-checked!

        } catch (ExpiredJwtException e) {
            throw new RuntimeException("Token expired");
        } catch (JwtException e) {
            throw new RuntimeException("Invalid token");
        }
    }

    // Usage:
    public static void main(String[] args) {
        String token = generateToken("user123", "ADMIN");
        System.out.println("Token: " + token);

        Claims claims = validateAndParse(token);
        System.out.println("User ID: " + claims.getSubject()); // user123
        System.out.println("Role: " + claims.get("role"));    // ADMIN
    }
Java Spring Boot Filter ‚Äî Intercept & Verify Every Request
@Component
public class JwtFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest req,
                                      HttpServletResponse res,
                                      FilterChain chain)
                                      throws IOException, ServletException {

        String header = req.getHeader("Authorization");

        if (header != null && header.startsWith("Bearer ")) {
            String token = header.substring(7); // strip "Bearer "

            Claims claims = JwtUtil.validateAndParse(token);

            // Set authentication in Spring Security context
            UsernamePasswordAuthenticationToken auth =
                new UsernamePasswordAuthenticationToken(
                    claims.getSubject(), null,
                    List.of(new SimpleGrantedAuthority(
                        (String) claims.get("role")
                    ))
                );
            SecurityContextHolder.getContext().setAuthentication(auth);
        }

        chain.doFilter(req, res);
    }
}
JWT vs Session-Based Auth
Aspect Session (Stateful) JWT (Stateless)
State stored where? Server memory / DB Client (token itself)
Scalability Hard (sticky sessions) Easy (any server can validate)
Revocation Easy (delete session) Hard (need blacklist)
Mobile/API friendly No (cookies) Yes (Authorization header)
Microservices Needs shared session store Each service validates independently
Common Interview Questions & Tips
‚úì Know This

"JWT is stateless ‚Äî the server doesn't need to store tokens. Validation just requires the secret key and checking expiry."

‚ö† Common Mistake

Don't say the payload is encrypted. It's only Base64URL encoded. Anyone can decode it ‚Äî signature just prevents tampering.

‚òÖ Bonus Point

Mention Refresh Tokens: short-lived access token (15 min) + long-lived refresh token (7 days) in httpOnly cookie. Best practice for production.

‚ö† Security Trap

The "alg: none" attack ‚Äî always specify and validate the algorithm. Never trust the header's alg claim blindly.

‚úì For Microservices

"In microservices, each service can validate the JWT independently without calling a central auth server ‚Äî that's the key advantage over session tokens."

‚òÖ Revocation Strategy

To revoke JWTs before expiry: maintain a Redis blacklist of invalidated jti (JWT ID) values. Check on every request.

Quick Cheat Sheet
TXT One-liners to remember
// JWT = Header.Payload.Signature (3 parts, dot-separated)
// Header  ‚Üí algo + type (alg + typ)
// Payload ‚Üí claims (sub, iat, exp, roles...)
// Sig     ‚Üí HMAC(base64Header + "." + base64Payload, secret)

// Payload is NOT encrypted, only encoded (anyone can read it)
// Signature verifies integrity (prevents tampering)

// Token sent as:  Authorization: Bearer <token>
// Stateless = no DB call to validate = fast + scalable

// HS256 = symmetric (one secret key for sign + verify)
// RS256 = asymmetric (private key signs, public key verifies)
//        ‚Üí RS256 preferred in microservices (each service has public key)

// Access Token  ‚Üí short-lived (15 min) ‚Äî sent in Authorization header
// Refresh Token ‚Üí long-lived (7 days)  ‚Äî stored in httpOnly cookie
