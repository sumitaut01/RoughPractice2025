Memory Zones Explained
🗂
Stack
Private per-thread. Holds method call frames with local variables and references. LIFO structure — frame pushed on call, popped on return. Very fast access.
Per-Thread · Auto-managed · LIFO
🌐
Heap
Shared across all threads. All objects and arrays live here. Garbage Collector automatically frees unreachable objects. Largest memory region.
Shared · GC managed · -Xmx
📚
Metaspace
Class-level metadata: bytecode, method signatures, static fields. Replaced PermGen in Java 8. Lives in native OS memory — grows dynamically.
Shared · Native OS · Java 8+
📍
PC Register
Each thread has its own tiny PC register storing the address of the currently executing JVM instruction. Undefined for native methods.
Per-Thread · JVM instruction ptr
🔧
Native Method Stack
Used when Java calls native (C/C++) methods via JNI. Managed by the OS, not JVM. Used by System.currentTimeMillis(), Object.hashCode() etc.
Per-Thread · OS managed · JNI

public class Example {
static int counter = 0; // ← Metaspace (static field)

void run() {
int x = 42; // ← Stack (local primitive)
String s = "hi"; // ← Stack (ref) → Heap String Pool
Example e = new Example(); // ← Stack (ref) → Heap (object)
counter++; // ← Metaspace (static access)
} // ← x, s, e refs popped from Stack
}
Common Memory Gotchas & Best Practices
⚠️ StackOverflowError — Infinite Recursion
Every method call pushes a new frame. Infinite recursion never pops, so the stack fills up.
void infinite() {
infinite(); // pushes new frame each call, never returns
} // → StackOverflowError
// Fix: ensure all recursion has a base case
void safeRecurse(int n) {
if (n == 0) return; // base case ✅
safeRecurse(n - 1);
}
⚠️ OutOfMemoryError — Heap Exhaustion
Creating objects faster than GC can collect them (memory leak) eventually exhausts the Heap.
List<byte[]> list = new ArrayList<>();
while (true) {
list.add(new byte[1024 * 1024]); // keeps adding 1MB objects
} // → OutOfMemoryError: Java heap space
⚠️ NullPointerException — Unboxing null Wrapper
When an Integer (or any wrapper) is null and you try to unbox it to a primitive, JVM calls .intValue() on null → NPE.
Integer obj = null; // wrapper can be null (on Heap)
int x = obj; // compiler: obj.intValue() → NPE! ❌
// Fix:
if (obj != null) { int x = obj; } // ✅
int x = (obj != null) ? obj : 0; // ✅
✅ Best Practice: Use primitives in hot loops
Wrapper types in loops cause repeated boxing/unboxing — creating thousands of short-lived Heap objects that increase GC pressure.
// BAD — Long wrapper boxing in every iteration
Long sum = 0L;
for (int i = 0; i < 1_000_000; i++) sum += i; // millions of Long objects!
// GOOD — pure primitive, no heap allocation
long sum = 0L; // ✅ Stack only
for (int i = 0; i < 1_000_000; i++) sum += i;
✅ Metaspace: Java 8+ PermGen is gone
Before Java 8, class metadata went into PermGen (fixed size, common source of "OutOfMemoryError: PermGen space"). Now it uses native OS memory and auto-expands.
// Java 7 and below:
// -XX:PermSize=64m -XX:MaxPermSize=256m ← Old way
// Java 8+:
// -XX:MetaspaceSize=64m -XX:MaxMetaspaceSize=256m ← New way ✅
