Section 01 — Interface Hierarchy

The Map Family Tree

«interface» Map<K,V> Root interface — key-value pairs, no duplicate keys
├─ «interface» SortedMap<K,V> extends Map — keys in natural/custom order
└─ «interface» NavigableMap<K,V> extends SortedMap — floor/ceiling/higher/lower navigation
└─ «abstract» AbstractMap<K,V> skeletal implementation of Map
└─ TreeMap<K,V> Red-Black tree · O(log n) · sorted · NavigableMap
├─ «interface» ConcurrentMap<K,V> extends Map — thread-safe atomic operations
└─ «interface» ConcurrentNavigableMap<K,V> ConcurrentMap + NavigableMap
└─ ConcurrentSkipListMap<K,V> Sorted + thread-safe · skip list
└─ ConcurrentHashMap<K,V> Segment/bucket locking · fail-safe iterator
├─ «abstract» AbstractMap<K,V> base class for most implementations
├─ HashMap<K,V> Hash table · O(1) avg · most used · fail-fast
└─ LinkedHashMap<K,V> HashMap + doubly-linked list · insertion order
├─ WeakHashMap<K,V> Weak references for keys — GC can collect
├─ IdentityHashMap<K,V> Uses == instead of .equals() for key comparison
└─ EnumMap<K extends Enum<K>,V> Enum keys only · array-backed · extremely fast
└─ Hashtable<K,V> Legacy · synchronized · no null keys/values · avoid

Section 02 — Implementations

Six Key Implementations

HashMap
java.util · since Java 1.2
"The workhorse — fast, unordered, everywhere"
OrderingNone
Null Keys1 allowed
Null ValuesMany allowed
Thread SafeNo
put / getO(1) avg
IteratorFail-Fast
Internals: Array of buckets (Node<K,V>[]). Each bucket is a linked list, upgraded to a Red-Black Tree when bucket size ≥ 8 (Java 8+). Default capacity 16, load factor 0.75. Rehashing doubles capacity when threshold exceeded.

Key comparison: Uses hashCode() to find bucket, then equals() to match key.
LinkedHashMap
java.util · since Java 1.4 · extends HashMap
"HashMap + insertion order memory"
OrderingInsertion / Access
Null Keys1 allowed
Null ValuesMany allowed
Thread SafeNo
put / getO(1) avg
IteratorFail-Fast
Extra: Maintains a doubly-linked list connecting entries in insertion order. Override accessOrder=true for LRU cache (access order). removeEldestEntry() enables automatic eviction — perfect for LRU caches.
TreeMap
java.util · since Java 1.2 · Red-Black Tree
"Sorted by key, navigation superpowers"
OrderingNatural / Comparator
Null KeysNo (throws NPE)
Null ValuesAllowed
Thread SafeNo
put / getO(log n)
IteratorFail-Fast
NavigableMap methods: firstKey(), lastKey(), floorKey(k), ceilingKey(k), headMap(k), tailMap(k), subMap(k1,k2). Keys must implement Comparable or provide a Comparator.
ConcurrentHashMap
java.util.concurrent · since Java 1.5
"Thread-safe, high-performance, no full lock"
OrderingNone
Null KeysNot allowed
Null ValuesNot allowed
Thread SafeYes (CAS+lock)
put / getO(1) avg
IteratorFail-Safe
Java 7: 16 segments, each independently locked. Java 8+: CAS (Compare-And-Swap) for put + synchronized only on individual bucket heads. Iterator works on a snapshot — modifications during iteration don't throw, but may not be reflected.
IdentityHashMap
java.util · since Java 1.4
"Reference equality, not .equals()"
Key Equality== (reference)
Null KeysAllowed
Thread SafeNo
Use CaseSerialization
Uses System.identityHashCode() instead of hashCode(). Two different String objects with the same value are treated as different keys. Used in serialization frameworks and topology-preserving graph algorithms.
EnumMap
java.util · since Java 1.5
"Enum keys only — blazing fast array-backed"
Key TypeEnum only
OrderingEnum ordinal
Null KeysNot allowed
PerformanceO(1) — fastest
Internally uses a plain array indexed by enum ordinal. No hashing at all — just array index access. Extremely compact and fast. Key null throws NullPointerException.

Section 04 — Iterator Safety

Fail-Fast vs Fail-Safe
Iterators

🔑 The modCount Mechanism — Root of Everything

Every structural modification to a HashMap (put new key, remove, clear) increments an internal counter called modCount. When you create an iterator, it records the current expectedModCount = modCount. On every next() call, the iterator checks: if (modCount != expectedModCount) → throw ConcurrentModificationException. This is fail-fast. Note: put on an existing key (just updating value) does NOT structurally modify, so it does NOT increment modCount.

⚡
Fail-Fast
Throws immediately on modification
Behaviour: Iterator checks modCount on every next() call. If the map was modified since iterator creation, throws ConcurrentModificationException immediately.

Which Maps:
  • HashMap
  • LinkedHashMap
  • TreeMap
  • Hashtable (legacy)
  • WeakHashMap
  • IdentityHashMap
Important: This is a best-effort guarantee — it's not guaranteed to throw in all concurrent cases. Do not rely on it for program correctness; it's only for detecting bugs.
🛡️
Fail-Safe
Never throws, may see stale data
Behaviour: Iterator operates on a snapshot/copy of the data. Modifications to the original map are NOT reflected in ongoing iteration. No exception thrown.

Which Maps:
  • ConcurrentHashMap
  • ConcurrentSkipListMap
  • Collections.synchronizedMap() — uses lock, not truly fail-safe
Trade-off: You may iterate over stale or incomplete data. New entries added during iteration may or may not be visible. Removed entries may still appear briefly.

Step-by-step: What happens when you modify during iteration?

⚡ FAIL-FAST — HashMap
1
HashMap has modCount = 5
2
Iterator created → records expectedModCount = 5
3
iterator.next() → checks modCount==5 ✅ returns entry
4
map.put("newKey", 99) → modCount becomes 6
5
iterator.next() → checks: 6 ≠ 5 → throws ConcurrentModificationException!
🛡️ FAIL-SAFE — ConcurrentHashMap
1
ConcurrentHashMap has entries {a=1, b=2, c=3}
2
Iterator created — reflects current state of segments
3
iterator.next() → returns "a=1" normally
4
map.put("d", 4) → map modified in another thread
5
iterator.next() → continues normally. "d=4" may or may not appear. No exception.

Safe removal during iteration (Fail-Fast maps)

Map<String,Integer> map = new HashMap<>(); map.put("a",1); map.put("b",2); map.put("c",3); // ❌ WRONG — throws ConcurrentModificationException for (String key : map.keySet()) { if (key.equals("b")) map.remove(key); // modifies map during iteration! } // ✅ CORRECT Way 1 — use Iterator.remove() Iterator<Map.Entry<String,Integer>> it = map.entrySet().iterator(); while (it.hasNext()) { Map.Entry<String,Integer> e = it.next(); if (e.getValue() < 2) it.remove(); // safe! uses iterator's remove } // ✅ CORRECT Way 2 — Java 8 removeIf (entrySet) map.entrySet().removeIf(e -> e.getValue() < 2); // ✅ CORRECT Way 3 — collect keys to remove, then remove List<String> toRemove = map.entrySet().stream() .filter(e -> e.getValue() < 2) .map(Map.Entry::getKey) .collect(Collectors.toList()); toRemove.forEach(map::remove);

Section 06 — When to Use What

Pick the Right Map

General Purpose
You need fast key-value storage, don't care about order, single-threaded.
→ HashMap
Preserve Insertion Order
Building an ordered cache, JSON serialization that should preserve field order.
→ LinkedHashMap
LRU Cache
Evict least-recently-used entries when capacity exceeded.
→ LinkedHashMap (accessOrder=true + removeEldestEntry)
Sorted Keys / Range Queries
Floor/ceiling queries, headMap, tailMap, subMap. e.g. routing tables, time-range queries.
→ TreeMap
Multi-threaded Access
Multiple threads reading/writing. High concurrency, no full lock needed.
→ ConcurrentHashMap
Enum Keys
Map with day-of-week, status codes, or any enum type as keys.
→ EnumMap (fastest possible)
Reference Identity Keys
Object graph traversal, serialization tracking — same value but different objects as separate keys.
→ IdentityHashMap
Avoid — Hashtable
Legacy synchronized map. Fully synchronized on every call — poor performance.
→ Use ConcurrentHashMap instead
Quick Decision Tree
Need thread-safety? YES → ConcurrentHashMap (no null keys/values) NO → Need ordering? SORTED BY KEY → TreeMap INSERTION ORDER → LinkedHashMap NO ORDER NEEDED → Key type? ENUM → EnumMap OTHER → HashMap ← default choice
